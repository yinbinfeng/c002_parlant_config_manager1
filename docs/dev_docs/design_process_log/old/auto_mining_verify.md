02_multi_agent_system_design.md设计的几点问题：
1. 目前的设计完全按照流程来设计了，没有仿照人类团队，
   比如，一开始需要有需求分析师，分析用户的输入；另外，每个节点需要有评审团；设计过程也可能设计讨论；
2. 每个agent需要调用的工具需要说明
3. 目前当作deep research已经作为工具封装，可直接调用
4. 每个子流程，都需要质检
5. 去重和合并不是简单的关键词去重，你需要语义去重
6. 另外，需要考虑规则/旅程/观测这些怎么构建联系
7. 需要考虑如果挖掘出2个旅程，有相同分支，如何合并

基于我的如下一些优化建议，请优化设计文档：

关于私域对话数据的处理，我有如下几个方案，帮我考虑一下那种方案最优（兼顾速度和准确度），然后补充进设计文档：
- 方案1： 对于私域对话数据的处理，你需要一通对话一通处理，否则上下文超长，
  之后利用大模型或向量比较，进行增删改查，为了提速，你可以采取类似graphRAG的增量式方案，否则，
  最后做融合，计算量很大
- 方案2： 利用deep research挖掘处理的内容，一通一通对话去判定，是否需要修改基于互联网的到的内容

关于deep research互联网搜索的注意事项：
- 虽然deep research包含深度和广度的搜索，但受上下文影响，搜索范围太广，效果仍然不佳，因此，
  搜索的话题仍需拆分成小的搜索方向，并发搜索，然后汇总结果

关于公域的互联网搜索和私域知识的融合方案，考虑如下那种方案最优，挑选一种：
- 方案1：先公域，再私域删减或补充
- 方案2：先私域，再公域聚焦方向补充
- 方案3：私域和公域并行处理，再汇总结果
