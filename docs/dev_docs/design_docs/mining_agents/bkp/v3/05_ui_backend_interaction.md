## 一、架构概述

### 1.1 设计背景

本系统采用 **Streamlit** 作为前端框架，前后端代码融合在 Python 中。系统支持两种调用方式：
1. **命令行/代码直接调用**：适合自动化、批处理场景
2. **UI 界面调用**：适合交互式、可视化场景

### 1.3 关键设计原则

| 原则 | 说明 |
|------|------|
| **统一核心** | UI 和命令行共享相同的 CoreEngine，确保逻辑一致性 |
| **状态驱动** | 所有交互基于状态机，支持断点续跑和状态恢复 |
| **会话隔离** | 每个任务有独立的 Session，避免状态污染 |
| **异步处理** | 耗时操作采用异步模式，UI 实时反馈进度 |

## 二、核心组件设计

### 2.1 SessionManager（会话管理层）

#### 2.1.1 职责
- 管理任务生命周期（创建、运行、暂停、恢复、完成）
- 维护会话上下文（用户输入、澄清回答、中间产物）
- 提供状态持久化（支持断点续跑）

#### 2.1.2 会话状态机

#### 2.1.3 会话数据结构

```yaml
session:
  session_id: "uuid-string"           # 会话唯一标识
  created_at: "2026-03-26T10:00:00Z"  # 创建时间
  updated_at: "2026-03-26T10:05:00Z"  # 更新时间
  
  # 用户输入
  input:
    description: "用户输入的需求描述"
    data_agent_file: "/path/to/file.json"  # 可选
    category: "软件服务"                      # 选中的分类
  
  # 当前状态
  status: "CLARIFICATION_REQUIRED"    # 当前状态
  current_step: 1                     # 当前步骤 (1-5)
  
  # 澄清相关
  clarification:
    generated_questions:              # Agent生成的问题列表
      - id: "q1"
        question: "问题内容"
        answer: ""                    # 用户回答（可选）
    user_added_questions:             # 用户新增的问题（★新增）
      - id: "uq1"
        question: "用户补充问题"
        answer: "用户回答"
  
  # 进度跟踪
  progress:
    overall_percent: 35               # 整体进度百分比
    current_action: "生成澄清问题"     # 当前正在执行的操作
    step_details:                     # 各步骤详情
      step1: { status: "completed", percent: 100 }
      step2: { status: "pending", percent: 0 }
      step3: { status: "pending", percent: 0 }
      step4: { status: "pending", percent: 0 }
      step5: { status: "pending", percent: 0 }
  
  # 输出产物
  output:
    config_package_path: "/output/agents/xxx/"  # 最终配置包路径
    intermediate_files: []                        # 中间产物文件列表
```

## 五、Streamlit 状态管理策略

### 5.1 Streamlit 执行模型

由于 Streamlit 是**脚本式执行模型**，每次用户交互都会重新运行整个脚本。因此需要特殊的状态管理策略：

```
Streamlit 执行模型:

用户操作 → 脚本重新运行 → 从st.session_state恢复状态 → 渲染UI
```

### 5.2 状态管理方案

| 状态项 | 用途 | 初始化值 |
|--------|------|----------|
| **session_id** | 当前会话标识 | None |
| **current_page** | 当前页面路由 | "input" |
| **status_poll_interval** | 状态轮询间隔(秒) | 2 |
| **show_add_question_form** | 是否显示新增问题表单 | False |

**状态轮询机制：**
- 使用 `st.empty()` 创建动态占位符
- 定时调用 `get_status()` 获取最新状态
- 状态变化时调用 `st.rerun()` 触发UI更新
- 轮询终止条件：状态为 CLARIFICATION_REQUIRED / COMPLETED / FAILED

### 5.3 澄清页面新增问题功能 ★

**界面布局设计：**

```
┌─────────────────────────────────────────────────────────┐
│                   澄清详情（可选）                       │
├─────────────────────────────────────────────────────────┤
│                                                         │

│  ┌─────────────────────────────────────────────────┐   │
│  │ q1: Agent生成的问题内容                          │   │

│  ┌─────────────────────────────────────────────────┐   │

│  │ ▼ 问题: 用户补充的问题...                        │   │

│  │ [保存] [取消]                                    │   │
```

**交互规则：**
1. **系统问题区域**：显示Agent生成的澄清问题，每个问题下方有回答输入框
2. **补充问题区域**：显示用户已新增的问题，使用折叠面板展示
3. **新增问题按钮**：点击后展开表单，包含问题输入框、回答输入框、保存/取消按钮
4. **自动保存**：用户填写回答后自动调用 `update_answer()` 保存
5. **表单验证**：保存时检查问题内容不能为空

## 七、优化建议

### 7.1 架构层面优化

| 优化点 | 说明 | 优先级 |
|--------|------|--------|
| **异步任务队列** | 引入Celery/RQ处理耗时任务，避免阻塞Streamlit | 高 |
| **WebSocket实时推送** | 替代轮询，使用WebSocket推送状态更新 | 中 |
| **会话持久化** | 使用SQLite/Redis存储会话，支持服务重启后恢复 | 中 |
| **文件存储分离** | 中间产物存储到对象存储(S3/OSS)，支持大文件 | 低 |

### 7.2 UI交互优化

| 优化点 | 说明 | 优先级 |
|--------|------|--------|
| **增量更新** | 只更新变化的部分，避免整个页面重绘 | 高 |
| **进度预估** | 基于历史数据预估剩余时间，提升用户体验 | 中 |
| **断点续跑UI** | 支持查看历史任务列表，恢复未完成任务 | 中 |
| **实时日志** | 显示Agent执行日志，便于调试和透明度 | 低 |

### 7.3 澄清功能优化 ★

| 优化点 | 说明 | 优先级 |
|--------|------|--------|
| **问题模板** | 提供常见问题模板，用户可快速选择添加 | 高 |
| **问题分类** | 用户新增问题可分类（业务/技术/其他） | 中 |
| **问题优先级** | 支持设置问题优先级，影响回答权重 | 低 |
| **富文本回答** | 支持Markdown格式的回答，便于结构化表达 | 低 |

### 7.4 结果展示优化
