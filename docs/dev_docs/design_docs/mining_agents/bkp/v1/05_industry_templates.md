### 6.1 业务维度识别 Prompt

```python
prompt = """
基于以下用户需求，识别需要覆盖的业务维度:

{initial_requirements}

请从以下维度进行分析:
1. **业务流程维度**: 售前/售中/售后？具体包含哪些流程？
2. **角色维度**: 涉及哪些角色？各自的职责是什么？
3. **场景维度**: 典型业务场景有哪些？高频场景是哪些？
4. **产品/服务维度**: 涉及哪些产品或服务类型？
5. **地域维度**: 是否有地域差异？不同地区的政策是否不同？
6. **合规维度**: 是否有特殊的监管要求或行业标准？

输出格式 (JSON):
{
    "business_flows": ["流程 1", "流程 2", ...],
    "roles": ["角色 1", "角色 2", ...],

}
"""
```

## 七、总结

### 7.1 各行业特点对比

| 行业 | 流程复杂度 | 合规重要性 | 典型查询数 | 关键维度 |
|------|----------|----------|----------|---------|
| **保险** | 高 | ⭐⭐⭐⭐⭐ | 24-32 | 推销、售后、理赔 |
| **电商** | 中 | ⭐⭐⭐ | 20-25 | 售前、售中、售后 |
| **医疗** | 高 | ⭐⭐⭐⭐⭐ | 25-30 | 预约、就诊、检查 |
| **金融** | 高 | ⭐⭐⭐⭐⭐ | 30-40 | 账户、贷款、理财 |
| **教育** | 中 | ⭐⭐⭐⭐ | 20-30 | 咨询、报名、学习 |

### 7.2 拆分原则总结

1. **业务流程优先**: 按售前→售中→售后主线拆分
2. **合规驱动**: 强监管行业 (金融、医疗、保险) 合规要求优先级最高
3. **场景导向**: 围绕典型业务场景组织知识
4. **角色视角**: 考虑不同角色的知识需求差异
5. **动态调整**: 根据用户具体需求灵活调整维度和权重

### 7.3 预期收益

- **覆盖面提升**: 相比固定拆分，多覆盖 30-50% 相关知识
- **精准度提升**: 搜索结果与业务场景的相关性提升 40%
- **可用性提升**: 结果按业务维度组织，后续使用效率提升 30%
- **加速比**: 5-8x (取决于查询数量和并发数)

### 8.4 通用设计原则

基于各行业的用户画像设计经验，总结以下通用原则:

#### (1) Tags 设计原则

| Tag 类别 | 示例 | 用途 |
|---------|------|------|
| **人口统计学** | `elderly`, `young_adult`, `student` | 识别基本属性 |
| **消费行为** | `price_sensitive`, `vip`, `impulse_buyer` | 指导销售策略 |
| **沟通偏好** | `phone_preference`, `detailed_explanation`, `fast_response` | 调整沟通方式 |
| **风险偏好** | `risk_averse`, `risk_tolerant` | 推荐合适产品 |
| **价值贡献** | `high_value`, `low_value`, `potential` | 资源分配优先级 |

#### (3) Guidelines 映射原则

| 群体特征 | 专属指南设计 | 优先级 |
|---------|-------------|-------|

| 风险规避 | 强调安全、避免高风险推荐 | 8-10 |
| 冲动购物 | 制造紧迫感、展示社交证明 | 5-7 |
