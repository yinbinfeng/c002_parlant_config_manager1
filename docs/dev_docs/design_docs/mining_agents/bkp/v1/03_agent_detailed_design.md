## 一、Coordinator Agent (项目经理)

### 1.1 职责定位

**人类角色**: 项目经理 (PM)

**核心职责**: 整体协调、进度控制、冲突仲裁

**必需性**: ⭐⭐⭐ 必需

**具体任务**:
1. 主持需求分析会议，介绍项目背景
2. 分配任务给各 Agent，设定优先级
3. 跟踪进度，确保各阶段按时完成
4. 仲裁 Agent 间的冲突和分歧
5. 资源调度和风险管理
6. 最终审批和交付决策

### 1.2 工具组合

| 工具 | 用途 | 使用频率 |
|------|------|----------|
| File System | 任务跟踪文件、进度记录 | 高频 |
| LLM API | 任务分配决策、冲突仲裁 | 高频 |
| Python Executor | 执行进度统计代码、生成报告 | 中频 |

### 1.3 关键算法

#### Step 1: 任务分配决策

基于项目需求和可用 Agent，制定合理的任务分配方案。考虑因素:
- 项目复杂度和规模
- 可用 Agent 列表 (考虑可选场景)
- Sprint 周期限制
- 关键路径任务识别
- 并行任务优先安排
- 预留评审时间

#### Step 2: 冲突仲裁

当 Agent 间出现冲突时:
1. 收集各方观点和证据
2. 基于仲裁原则进行公正裁决
3. 形成行动项清单并跟踪执行

**仲裁原则**:
- 以项目整体利益为重
- 尊重专业意见 (DomainExpert, CustomerAdvocate)
- 考虑时间和资源约束
- 优先保证关键路径

### 1.4 输入输出规范

**输入**:
- 项目章程 (领域关键词、部署场景、时间线)
- 客户需求描述

**输出**:
- 项目计划 (Sprint 划分、任务分配、关键路径)
- 进度报告
- 冲突仲裁结果

## 二、RequirementAnalyst Agent (需求分析师)

### 2.1 职责定位

**人类角色**: 需求分析师 (BA)

**核心职责**: 理解用户需求、澄清模糊点、定义范围

**必需性**: ⭐⭐⭐ 必需

**具体任务**:
1. 主持客户需求会议，记录需求
2. 提出澄清问题，消除歧义
3. 需求分解和结构化
4. 定义项目边界和优先级
5. 形成需求规格说明书

### 2.2 工具组合

| 工具 | 用途 | 使用频率 |
|------|------|----------|
| LLM API | 需求解析、问题生成 | 高频 |
| File System | 需求文档管理 | 高频 |
| Code Generator | 生成需求模板、结构化文档 | 中频 |

### 2.3 关键算法

#### Step 1: 澄清问题生成

分析客户需求，识别模糊点，从以下维度生成结构化问题清单:
1. **功能性需求**: 具体需要实现什么功能？
2. **非功能性需求**: 性能、可用性、安全性要求？
3. **用户群体**: 目标用户是谁？有什么特点？
4. **业务场景**: 在什么场景下使用？
5. **成功标准**: 如何判断需求已满足？
6. **约束条件**: 技术、时间、资源限制？
7. **依赖关系**: 依赖哪些外部系统或数据？

#### Step 2: 需求结构化

将客户回答转换为结构化的需求规格说明书，包含:
- 项目概述 (背景、目标、范围)
- 功能性需求 (Use Case 形式)
- 非功能性需求 (性能指标、安全要求)
- 用户特征 (目标用户、使用场景)
- 成功标准 (可衡量的验收条件)
- 约束条件 (技术栈、时间、预算)
- 依赖关系 (外部系统、数据源)
- 风险因素 (可能的风险和应对策略)

### 2.4 输入输出规范

**输入**:
- 客户初始需求描述
- 项目上下文 (行业、公司规模、现有系统)

**输出**:
- 需求文档 v2.0(评审通过)
- 澄清问题日志

## 三、DomainExpert Agent (领域专家)

### 3.1 职责定位

**人类角色**: 领域专家 (SME)

**核心职责**: 提供行业知识、最佳实践、合规要求

**必需性**: ⭐⭐ 推荐 (无公开资料时可省略)

**具体任务**:
1. 使用 Deep Research 搜索行业资料
2. 整理行业通用术语和定义
3. 收集最佳实践和合规要求
4. 分析竞品配置特点
5. 与 DataAnalyst 对比公开 vs 私域知识

### 3.2 工具组合

| 工具 | 用途 | 使用频率 |
|------|------|----------|
| Deep Research | 联网搜索行业资料 | 高频 |
| LLM API | 信息提取、知识整理 | 高频 |
| File System | 知识库管理 | 中频 |

### 3.3 工作流程

**关键优化**: 采用"基于业务维度的智能拆分 + 并发搜索"策略

1. **制定搜索策略**: 基于领域关键词和业务目标
2. **生成批量搜索查询**: 拆分为 10-50 个小查询
3. **并发执行搜索**: 使用信号量控制并发数 (5-10)
4. **信息提取**: 从搜索结果中提取术语、流程、最佳实践、合规要求
5. **增量保存**: 保存到公开知识库
6. **去重和融合**: 使用 Embedding 去重，交叉分析
7. **质量检查**: 评估来源权威性和时效性

**并行优化点**: 
- 4 个搜索分支完全独立，可并行执行，加速比 1.8x
- 每个分支内批量搜索可并发，加速比 2.5x
- 总体理论加速比：4.5x

### 3.4 输入输出规范

**输入**:
- 领域关键词
- 搜索深度 (deep/standard/shallow)
- 目标行业

**输出**:
- 公开知识库 (术语、流程、最佳实践、合规要求)
- 搜索报告 (来源数量、质量评分)

## 四、DataAnalyst Agent (数据分析师)

### 4.1 职责定位

**人类角色**: 数据分析师

**核心职责**: 分析对话数据、发现模式、统计洞察

**必需性**: ⚠️ 依赖私域数据 (无私域数据时不需要)

**具体任务**:
1. 加载和预处理对话日志
2. 对话聚类和场景识别
3. 从对话中提取业务流程
4. 统计分析和模式发现
5. 与 DomainExpert 的公开知识对比融合
6. **向 UserPortraitMiner 提供结构化对话数据**

### 4.2 工具组合

| 工具 | 用途 | 使用频率 |
|------|------|----------|
| LLM API | 对话分析、场景识别 | 高频 |
| File System | 对话数据读写 | 高频 |
| Python Executor | 数据挖掘、统计分析 | 中频 |

### 4.3 工作流程

**关键优化**: 采用"分治策略"

1. **单通对话独立处理**: 避免超长上下文，每次聚焦一通对话
2. **事件提取**: 使用 LLM 识别对话中的关键事件和意图
3. **场景聚类**: 使用 Embedding + DBSCAN 自动发现场景簇
4. **流程挖掘**: 使用 Heuristic Miner 发现频繁路径和分支
5. **检查点机制**: 每 10 个批次保存一次，防止中断丢失
6. **后期聚合**: 所有对话处理完成后，合并统计结果
7. **用户特征标注**: 为每通对话标注用户 Tags(如价格敏感、VIP、投诉倾向)

**并行优化点**: 单通对话完全并行处理，理论加速比=对话数量

### 4.4 输入输出规范

**输入**:
- 对话日志路径
- 时间范围
- 最小对话数量

**输出**:
- 私域知识库 (场景、流程、统计信息)
- 对话分析报告
- **标注用户特征的对话数据集 **(供 UserPortraitMiner 使用)

## 五、UserPortraitMiner Agent (用户画像分析师)

### 5.1 职责定位

**人类角色**: 用户画像分析师 / 客户洞察专家

**核心职责**: 从对话数据中挖掘用户特征、分群聚类、生成典型用户画像

**必需性**: ⚠️ 依赖私域数据 (需要个性化服务时必需)

**具体任务**:
1. 从 DataAnalyst 接收标注用户特征的对话数据
2. 提取用户行为模式和偏好特征
3. 基于 Embedding 和聚类的用户分群
4. 生成典型用户画像 (Personas)
5. 映射到 Parlant 配置 (Tags, Variables, Guidelines, Journeys)
6. 设计群体特定的专属指南
7. 与 CustomerAdvocate 协作验证画像合理性

### 5.2 工具组合

| 工具 | 用途 | 使用频率 |
|------|------|----------|
| LLM API | 用户特征抽象、画像生成 | 高频 |
| Embedding Service | 用户行为向量化、相似度计算 | 高频 |
| Python Executor | 聚类算法 (DBSCAN/K-Means)、统计分析 | 高频 |
| File System | 画像数据存储 | 中频 |

### 5.3 工作流程

**关键优化**: 采用"增量式用户特征聚合 + 分群聚类"

#### Step 1: 用户特征提取

从 DataAnalyst 提供的标注对话中提取:
- **显式标签**: VIP、价格敏感、老年客户、投诉倾向等
- **隐式模式**: 决策时长、询问频率、渠道偏好等
- **元数据规则**: 消费金额、年龄、购买频次等数值范围

**示例**:
```
对话标注：用户多次询问折扣、使用优惠券、对比竞品
→ 提取特征:
   - Tags: ["price_sensitive", "coupon_user"]
   - Behavior: "频繁询问价格和折扣"
   - Metadata: {"coupon_usage_rate": ">0.5"}
```

#### Step 2: 增量聚合统计

维护全局用户特征统计表，实时聚合:
- 每个 Tag 独立统计出现频次、行为模式、对话摘要
- 每处理一通对话，实时更新对应 Tag 的统计数据
- 每 10 个批次保存一次检查点，防止中断丢失

**统计表示例**:
```
global_feature_stats = {
    "price_sensitive": {
        "count": 350,  # 35% 的对话
        "behaviors": ["频繁询问折扣", "对比竞品价格", ...],
        "dialogues": ["对话摘要 1", "对话摘要 2", ...],
        "metadata_examples": [{"price_sensitivity_score": 0.8}, ...]
    },
    "vip_customers": {...}
}
```

#### Step 3: 用户分群聚类

使用无监督学习自动发现用户群体:

**算法流程**:
1. **向量化**: 将用户对话序列转换为 Embedding 向量
2. **构建特征矩阵**: 基于 Tags 频次、行为模式、元数据构建数值特征
3. **标准化处理**: 消除不同特征的尺度差异
4. **聚类分析**: 使用 DBSCAN 或 K-Means 发现自然分组
5. **特征提取**: 为每个簇提取代表性特征和高频 Tags

**算法选择**:
- **DBSCAN**: 无需预设簇数，能识别任意形状，适合探索性分析
- **K-Means**: 需要预设 K 值，计算效率高，适合已知分群数量的场景

#### Step 4: 典型用户画像生成

LLM 基于聚类结果生成拟人化画像:

**输入信息**:
- 簇名称 (如"价格敏感群体")
- 簇内高频 Tags
- 代表性行为模式 (Top 5)
- 典型对话片段 (Top 3)

**生成要素**:
1. **姓名**: 拟人化命名 (如"精明消费者小王")
2. **人口统计学特征**: 年龄段、职业类型、收入水平、教育背景
3. **核心目标**: 该用户使用产品或服务的主要目标
4. **主要痛点**: 2-3 个最突出的痛点
5. **行为模式**: 3-5 条典型行为特征
6. **典型对话**: 3-5 句该用户常说的典型话语
7. **Parlant 映射**: Tags、Variables、Journeys、Guidelines

#### Step 5: 专属指南映射

为每个用户分群设计专属 Guidelines:

**设计原则**:
- **条件触发**: 基于 Observations 检查客户 Tags
- **个性化动作**: 针对群体特征提供差异化服务
- **优先级设置**: 群体特定指南优先级高于通用指南

**示例**:
- **VIP 客户**: "提供专属 9 折优惠"、"跳过排队，优先安排"
- **价格敏感客户**: "强调节省金额"、"主动提供优惠券"
- **老年客户**: "使用简单易懂语言"、"提供电话人工协助"

#### Step 6: 输出用户画像配置

生成符合 Parlant 格式的用户画像配置文件:

**配置结构**:
```json
{
  "agent_id": "medical_customer_service_agent_001",
  "user_segments": [
    {
      "segment_id": "vip_customers",
      "segment_name": "VIP 客户",
      "definition": {
        "tags": ["vip", "high_value"],
        "metadata_rules": {"total_purchase_amount": ">10000"}
      },
      "behavior_patterns": [...],
      "special_guidelines": [...],
      "custom_variables": {...}
    }
  ],
  "personas": [
    {
      "persona_id": "persona_vip_001",
      "persona_name": "高端商务人士张总",
      "demographics": "40-50 岁，企业高管，年收入 200 万+",
      "parlant_mapping": {...}
    }
  ]
}
```

### 5.4 输入输出规范

**输入**:
- DataAnalyst 提供的标注对话数据集
- 用户分群数量建议 (可选，由算法自动决定)
- 行业特定用户特征模板 (可选)

**输出**:
- **用户画像库 v1.0**,包含:
  - 用户分群定义 (Segments)
  - 典型用户画像 (Personas)
  - 专属指南映射 (Special Guidelines)
  - 自定义变量定义 (Custom Variables)
- 用户画像分析报告 (含统计信息、聚类结果、代表性对话)

### 5.5 与 CustomerAdvocate 协作

CustomerAdvocate 负责验证画像的合理性:
1. **模拟测试**: 基于画像生成测试对话，验证行为一致性
2. **体验反馈**: 从用户角度评估画像是否真实反映需求
3. **改进建议**: 提出画像细化和补充意见

**协作流程**:
```
UserPortraitMiner 生成初版画像
         ↓
CustomerAdvocate 模拟测试和评估
         ↓
收集反馈和痛点
         ↓
UserPortraitMiner 迭代优化画像
         ↓
评审团评审通过
```

## 八、JourneyBuilder Agent (旅程构建师)

### 8.1 职责定位

**人类角色**: 旅程构建师

**核心职责**: 构建 Journey 状态机、绑定规则和工具

**必需性**: ⭐⭐⭐ 必需

**具体任务**:
1. 将流程映射为状态机
2. 设计状态转换逻辑
3. 绑定 Observations 和 Guidelines
4. 集成 Tool 调用 (如有)
5. 验证状态机 soundness

### 8.2 工具组合

| 工具 | 用途 | 使用频率 |
|------|------|----------|
| LLM API | 状态映射、转换设计 | 高频 |
| Embedding Service | 分支合并检测 | 中频 |
| File System | 配置保存 | 中频 |

### 8.3 工作流程

1. **映射流程到状态**: 将业务流程转换为状态机
2. **设计初始状态**: 定义入口
3. **设计 Chat 状态**: 对话交互状态
4. **设计 Tool 状态**: 工具调用状态
5. **定义转换**: 状态间的转换逻辑
6. **添加条件**: 转换条件和约束
7. **绑定规则**: 关联 Observations 和 Guidelines
8. **绑定工具**: 集成 Tool 调用
9. **验证状态机**: 检查可达性、无环性、无孤立状态

### 8.4 输入输出规范

**输入**:
- 业务流程图
- 规则集
- 工具包

**输出**:
- Journey 配置 v1.0
- 状态机验证报告
