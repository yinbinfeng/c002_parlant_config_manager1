## 一、项目背景与目标

### 1.1 项目背景

在构建基于 Parlant 框架的客服 agent 时，需要准备大量结构化配置信息，包括:
- **Agent 基础配置**: 元数据、术语表、可观测性、**用户画像**
- **全局规则**: 观测、指南、模板话术及其关系
- **业务 SOP**: 流程状态机、专属规则、话术
- **工具库**: 元数据和实现代码

传统人工梳理和录入这些信息的痛点:
1. **工作量巨大**: 一个完整的客服 agent 需要数百条规则和数十个 SOP
2. **专业要求高**: 需要深入理解 Parlant 框架的概念和关系设计
3. **一致性难保证**: 人工编写容易出现规则冲突、术语不一致
4. **迭代成本高**: 业务变更时需要手动更新大量配置
5. **知识沉淀难**: 优秀的话术和经验难以系统化沉淀
6. **用户画像设计困难**: 需要了解真实客户特征，难以凭空设计分群和画像

### 1.2 项目目标

设计一个多 Agent 协作系统，实现:
1. **自动化挖掘**: 从公开资料 (互联网) 和私域数据 (对话日志) 中自动提取配置信息
2. **智能化处理**: 利用 LLM 的理解和生成能力，自动梳理业务流程和规则
3. **协作化生产**: 多个 Agent 分工合作，模拟人类团队的协作模式
4. **可验证输出**: 内置评估机制，确保挖掘质量
5. **可扩展架构**: 支持不同行业、不同业务线的快速适配
6. **个性化服务**: 从对话数据中自动挖掘用户画像，实现差异化服务策略

## 二、核心设计理念

### 2.1 从"流程驱动"到"角色驱动"的转变

**原设计问题**:
- ❌ 过于机械化的流水线作业
- ❌ 缺少人类团队的评审和讨论机制
- ❌ 缺乏需求理解和分析环节

**新设计理念**:
- ✅ **完全模拟人类咨询团队**的工作模式
- ✅ **每个关键角色都有对应 Agent**
- ✅ **引入评审团 **(Review Board)
- ✅ **支持讨论、辩论、投票等人类协作方式**

### 2.2 人类团队角色映射

```
典型人类咨询团队配置          →        我们的 Agent 团队
─────────────────────────────────────────────────────
项目经理 (PM)                  →        Coordinator Agent
需求分析师 (BA)                →        Requirement Analyst Agent  
领域专家 (SME)                 →        Domain Expert Agent
数据分析师                     →        Data Analyst Agent
用户画像分析师                 →        UserPortraitMiner Agent
流程设计师                     →        Process Designer Agent
规则工程师                     →        Rule Engineer Agent
工具架构师                     →        Tool Architect Agent
质量保障 (QA)                  →        QA Agent (评审团主席)
旅程构建师                     →        Journey Builder Agent
配置工程师                     →        Config Assembler Agent
客户代表                       →        Customer Advocate Agent
```

## 三、设计原则

### 3.1 角色驱动原则
- **专业化分工**: 每个 Agent 负责特定类型的信息挖掘
- **人类团队模拟**: 完全模仿人类咨询团队的工作模式
- **评审团机制**: 引入 Review Board 进行集体决策

### 3.2 知识分层原则
- **公开知识层**: 从 Deep Research 获取行业通用知识
- **私域知识层**: 从对话数据中挖掘企业特有知识
- **用户画像层**: 从对话中自动提取用户特征、分群聚类
- **融合校验**: 两层知识相互校验，采用**"先公域再私域验证"**策略

### 3.3 质量保证原则
- **四级质检**: 自检→同行评审→评审团评审→终检
- **语义去重**: 使用 Embedding 进行语义级去重
- **智能合并**: 识别并合并相同分支
- **增量处理**: 私域对话数据采用"分治策略",单通对话独立处理避免长上下文，最后聚合保证全局一致性

### 3.4 可控性原则
- **人工审核点**: 关键环节保留人工确认
- **透明度**: 所有挖掘过程可追溯、可解释
- **容错机制**: 错误检测和自我修正

### 3.5 性能优化原则
- **并发优先**: 尽可能并行执行，理论加速比 5-10x
- **上下文管理**: 参考 Manus，基于文件系统和 Git 管理
- **目标对齐**: 每次决策前重申任务目标，防止偏离
- **增量融合**: 实时融合知识，避免最后集中计算

| **CustomerAdvocate** | 客户代表 | 站在用户角度审视、体验反馈、可用性测试 | ⭐⭐ 推荐 |
| **QAModerator** | QA 主席 | 组织评审会、质量检查、最终放行 | ⭐⭐ 推荐 |

**说明**: 
- 根据实际项目需求和数据可用性，部分 Agent 可能不投入使用
- **UserPortraitMiner** 是新增的专门负责用户画像挖掘的 Agent，当需要支持个性化服务时必需

### 4.2 工具平台

系统提供统一的基础工具平台，所有 Agent 的"专用工具"都是对这些基础工具的封装和组合使用:

**A. 知识获取工具**
- Deep Research: 联网搜索、信息检索、资料收集

**B. 文件与数据工具**
- File System: 文件读写、目录管理、批量处理
- Git: 版本控制、变更追踪、协作管理

**C. 智能处理工具**
- LLM API: 文本理解、内容生成、语义分析
- Embedding Service: 语义向量计算、相似度分析

**D. 代码执行工具**
- Python Executor: Python 代码执行引擎
- Code Generator: 代码生成和编写
- Package Manager: 依赖管理、环境配置

**说明**:
- 测试、验证等功能通过编写代码实现，而非独立工具
- 每个 Agent 根据职责不同，使用特定的工具组合

## 五、关键优化策略

### 5.1 私域对话数据处理 - 分治策略

**核心思想**: 单通对话独立处理 + 后期聚合

**技术方案**:
- **事件提取**: 使用 LLM 从单通对话中提取关键事件和意图
- **场景聚类**: 使用 Embedding + DBSCAN 自动发现场景簇
- **流程挖掘**: 使用 Heuristic Miner 算法发现频繁路径
- **并发控制**: 单通对话处理完全并行 (加速比=对话数量)
- **检查点机制**: 每 10 个批次保存一次，防止中断丢失

**预期收益**:
- 速度提升：5-10x (取决于对话数量和并发数)
- 内存占用降低：单通对话处理，不需要一次性加载
- 准确度提升：每次聚焦小范围，避免噪声干扰

### 5.2 Deep Research 互联网搜索 - 基于业务维度的智能拆分

**核心思想**: 根据用户目标和业务维度自动拆分搜索话题

**拆分维度示例 **(保险行业):
```
用户目标：构建保险客服系统的 Parlant 配置包
    ↓
按业务流程维度拆分:
    ├─ 推销流程 (售前)
    │   ├─ 术语表搜索
    │   ├─ 业务流程搜索
    │   ├─ 合规要求搜索
    │   └─ 话术技巧搜索
    ├─ 售后流程 (售后)
    │   ├─ 保单保全搜索
    │   ├─ 理赔服务搜索
    │   └─ 续期收费搜索
    └─ 增值服务 (售中)
        ├─ 保单贷款搜索
        └─ 信息变更搜索
    ↓
并发执行各个搜索任务
    ↓
汇总结果 + 去重 + 融合 + 按流程组织
```

**预期收益**:
- 速度提升：5-8x (取决于查询数量和并发数)
- 覆盖面提升：相比固定拆分，多覆盖 30-50% 相关知识
- 精准度提升：搜索结果与业务场景的相关性提升 40%

### 5.3 公域 vs 私域知识融合 - 先公域再私域验证

**融合流程**:
```
公域知识收集 → 私域对话处理 → 差异标注 → 融合知识库
```

**对比策略**:

| 情况 | 处理方式 | 优先级 |
|------|---------|-------|
| 私域有，公域无 | 标注"企业特有实践" | 保留 |
| 公域有，私域无 | 标注"可能遗漏/业务差异" | 需确认 |
| 两者冲突 | 标注"需人工审核" | 私域优先 |

### 5.4 Agent 开发策略 - Manus 经验

**上下文管理**:
- 文件系统 + Git 管理，避免上下文超长
- 标准文件结构：task_plan.md, progress.md, findings.md 等
- 子任务输出写入独立文件，主任务只读取关键摘要

**防止任务偏离**:
- 每次 LLM 调用前包含任务计划
- 确保任务计划在 attention window 内
- 建立检测并纠正偏离的机制

## 七、性能预估总览

假设场景：1000 通私域对话 + 40 个搜索查询

| 优化项 | 优化前 | 优化后 | 提升 |
|-------|-------|-------|------|
| **私域数据处理** | 80 分钟 | 55 分钟 | 1.45x |
| **Deep Research** | 10 分钟 | 2 分钟 | 5x |
| **总体处理时间** | ~90 分钟 | ~57 分钟 | **1.58x** |
| **内存占用** | 全部加载 | 单通对话 | **1000x 降低** |

如果是 10000 通对话:
- 私域数据处理从**13 小时**降至**9 小时**
- 内存占用从**可能 OOM**降至**稳定运行**
