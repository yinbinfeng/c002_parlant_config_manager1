## 一、私域对话数据处理 - 分治策略

### 1.1 方案选择：分而治之

**选定方案**: 方案 1 - 分治策略 (单通独立处理 + 后期聚合)

**理由**:
- ✅ **速度优势**: 单通对话独立处理，完全并行，加速比=对话数量
- ✅ **准确度优势**: 每次聚焦小范围，避免长上下文干扰
- ✅ **内存友好**: 不需要一次性加载所有对话
- ✅ **技术简单**: 无需引入图数据库，降低运维成本
- ✅ **成熟算法**: 采用流程挖掘和聚类领域的经典算法

### 1.2 核心设计

```
私域对话数据处理流程:

Step 1: 单通对话独立处理 (避免超长上下文)
  ┌─────────────┐
  │ 对话日志 1   │→ 预处理 → 事件提取 → 场景识别
  ├─────────────┤
  │ 对话日志 2   │→ 预处理 → 事件提取 → 场景识别
  ├─────────────┤
  │ 对话日志 N   │→ 预处理 → 事件提取 → 场景识别
  └─────────────┘
       ↓
Step 2: 场景聚类 (Embedding + DBSCAN)
  - 将所有对话的 Embedding 输入聚类算法
  - 自动发现场景簇 (无需预设簇数量)
  - 计算每个簇的代表性对话
       ↓
Step 3: 流程挖掘 (Heuristic Miner)
  - 每个场景簇内的事件序列分析
  - 发现频繁路径和分支模式
  - 生成 BPMN 流程图
       ↓
Step 4: 与公域知识对比融合
  - 输入：DomainExpert 提供的公域知识库
  - 对比策略:
    • 标注差异：私域有但公域无 → 企业特有实践
    • 标注缺失：公域有但私域无 → 可能遗漏或业务不同
    • 标注冲突：两者不一致 → 标记需人工审核
  - 输出：融合知识库 (标注来源和置信度)
```

**并行优化效果**:
- 单通对话处理完全并行，理论加速比 = 对话数量 (如 1000 通对话=1000x)
- 场景聚类和流程挖掘可并行处理多个簇
- 总体理论加速比：5-10x (取决于对话数量)

### 1.3 关键实现要点

#### 事件提取模块

**核心机制**:
- 使用 LLM 从对话轮次中提取关键事件
- 识别用户意图 (咨询、办理、投诉等)
- 提取槽位信息 (时间、地点、对象等)

**示例**:
```
对话："我想预约挂号"
→ 事件：挂号申请
→ 意图：预约就诊
→ 槽位：{科室：未指定，时间：未指定}
```

#### 场景聚类模块

**算法选择**: DBSCAN (Density-Based Spatial Clustering)

**优势**:
- 无需预设簇数量 (自动发现场景数)
- 能识别任意形状的簇
- 对噪声点鲁棒 (异常对话不影响)

**参数建议**:
- eps (邻域半径): 0.3-0.5 (基于 Embedding 维度调整)
- min_samples (最小样本数): 5-10 (保证每个场景有统计意义)

#### 流程挖掘模块

**算法选择**: Heuristic Miner (启发式流程挖掘)

**优势**:
- 支持并发路径识别
- 支持循环结构
- 对噪声鲁棒 (过滤低频路径)
- 输出 BPMN/Petri 网标准格式

**工具推荐**: PM4Py (Python 库)

#### 检查点机制

- 每处理 10 个批次保存一次中间结果
- 防止中断导致的数据丢失
- 支持断点续传

### 1.4 性能预估

假设条件:
- 对话数量：1000 通
- 单通对话处理时间：30 秒 (包括 LLM 调用和 Embedding 计算)
- 并发数：10

**传统批量处理方案**:
```
总时间 = 1000 通 × 30 秒 ÷ 10 并发 = 50 分钟
最后聚合时间 = 约 30 分钟 (聚类 + 流程挖掘)
总计：~80 分钟
```

**分治策略方案**:
```
总时间 = 1000 通 × 30 秒 ÷ 10 并发 = 50 分钟
场景聚类时间 = ~3 分钟 (批量处理，非常快)
流程挖掘时间 = ~2 分钟 (每个场景独立，可并行)
总计：~55 分钟

加速比：80/55 = 1.45x
```

**如果对话数量为 10000 通**:
- 传统方案：~800 分钟 (13 小时)
- 增量方案：~550 分钟 (9 小时)
- 加速比：**1.45x**, 且内存占用大幅降低

## 二、Deep Research 互联网搜索 - 基于业务维度的智能拆分

### 2.1 核心问题

虽然 Deep Research 包含深度和广度的搜索，但受上下文影响，搜索范围太广效果仍然不佳。

### 2.2 优化方案：基于业务维度的智能拆分 + 并发搜索

**核心思想**: 根据用户目标和业务维度自动拆分搜索话题，而非简单的固定拆分。

**拆分维度示例 **(以保险行业为例):

```
用户目标：构建保险客服系统的 Parlant 配置包
    ↓
Step 1: 按业务流程维度拆分
    ├─ 推销流程 (售前)
    │   ├─ 客户开发
    │   ├─ 需求分析
    │   ├─ 产品推荐
    │   └─ 促成签单
    ├─ 售后流程 (售后)
    │   ├─ 保单保全
    │   ├─ 理赔服务
    │   ├─ 续期收费
    │   └─ 投诉处理
    └─ 增值服务 (售中)
        ├─ 保单贷款
        ├─ 信息变更
        └─ 生存金领取
    ↓
Step 2: 每个流程再按知识类型拆分
    推销流程 → {
        术语表搜索 (10 个查询),
        最佳实践搜索 (15 个查询),
        合规要求搜索 (8 个查询),
        话术技巧搜索 (12 个查询)
    }
    售后流程 → {
        术语表搜索 (12 个查询),
        理赔流程搜索 (10 个查询),
        合规要求搜索 (15 个查询),
        典型案例搜索 (8 个查询)
    }
    ↓
Step 3: 并发执行各个搜索任务
    ↓
Step 4: 汇总结果 + 去重 + 融合 + 按流程组织
```

### 2.3 实施要点

#### (1) 业务维度识别

在开始搜索前，先与用户确认业务维度:

**识别维度**:
1. **业务流程维度**: 售前/售中/售后？具体包含哪些流程？
2. **角色维度**: 涉及哪些角色 (如销售、客服、理赔员)?
3. **场景维度**: 典型业务场景有哪些？
4. **产品维度**: 涉及哪些产品或服务类型？
5. **地域维度**: 是否有地域差异 (如大陆/台湾/香港)?

#### (2) 动态优先级调整

不是所有查询都同等重要，根据业务特点动态调整优先级:

**规则示例**:
- 合规要求 > 业务流程 > 最佳实践 (保险行业)
- 医疗流程 > 销售技巧 (医疗行业)
- 安全规范 > 用户体验 (金融行业)

#### (3) 并发控制策略

- **分层并发**: 
  - 第一层：不同业务目标之间并发 (如推销流程 vs 售后流程)
  - 第二层：同一目标内不同知识类型并发
  
- **资源分配**: 高优先级的业务目标分配更多并发配额

#### (4) 结果组织与融合

搜索结果按业务维度组织，便于后续使用:
- 按业务目标分组
- 按知识类型分组
- 提取跨目标的通用知识

### 2.4 完整示例：保险行业

**输入**:
```
domain = ["保险", "人身保险", "寿险"]
business_goals = [
    "推销流程",
    "售后流程", 
    "理赔流程",
    "保单保全流程"
]
```

**生成的搜索查询结构**:
- **推销流程**: 术语表、业务流程、合规要求、话术技巧
- **售后流程**: 术语表、业务流程、合规要求、典型案例
- **理赔流程**: 术语表、业务流程、合规要求、典型案例
- **保单保全流程**: 术语表、业务流程、合规要求

**预期效果**:
- **覆盖面**: 4 个业务目标 × 平均 8 种知识类型 = 32 个查询方向
- **精准度**: 每个查询都针对具体业务场景
- **可复用性**: 结果按业务维度组织，便于后续 Agent 使用

### 2.5 性能预估

假设条件:
- 业务目标数量：4 个 (如推销、售后、理赔、保全)
- 每目标知识类型：8 种
- 总查询数：32 个
- 单查询时间：15 秒
- 并发数：5

**传统串行搜索**:
```
总时间 = 32 × 15 秒 = 480 秒 = 8 分钟
```

**并发搜索 **(按业务维度分组):
```
总时间 = 32 × 15 秒 ÷ 5 并发 = 96 秒 = 1.6 分钟
加速比：5x

额外收益:
- 结果按业务维度组织，后续使用效率提升 30%
- 优先级高的业务目标获得更多资源，质量提升
```

**如果查询数为 100 个**:
- 串行：100 × 15 秒 = 25 分钟
- 并发：100 × 15 秒 ÷ 8 并发 = 3.1 分钟
- 加速比：**8x**

## 三、公域与私域知识融合方案

### 3.1 方案选择：先公域再私域验证

**选定方案**: 方案 1 - 先公域 → 私域删补

**理由**:
- ✅ **认知合理性**: 先建立行业基准，再用企业数据验证
- ✅ **效率优势**: 公域提供框架，私域只需填充和修正
- ✅ **质量保证**: 有行业标杆作为参照，容易发现异常

### 3.2 融合流程

```
Step 1: DomainExpert 收集公域知识
  - 行业通用术语
  - 标准业务流程
  - 最佳实践
  - 合规要求
  ↓
Step 2: DataAnalyst 处理私域对话
  - 增量式提取场景和流程
  - 实时与公域对比
  ↓
Step 3: 差异标注与融合
  | 对比结果 | 处理方式 |
  |---------|---------|
  | 私域有，公域无 | 标注为"企业特有实践"，保留 |
  | 公域有，私域无 | 标注为"可能遗漏"或"业务差异"，需确认 |
  | 两者冲突 | 标注为"需人工审核"，优先采用私域 (更贴近实际) |
  ↓
Step 4: 生成融合知识库
  - 每个知识点标注来源 (public/private)
  - 标注置信度 (高/中/低)
  - 标注需人工审核项
```

### 3.3 对比维度

| 维度 | 公域知识 | 私域知识 | 融合策略 |
|------|---------|---------|---------|
| **覆盖面** | 广而浅 | 窄而深 | 公域为骨架，私域填充细节 |
| **准确性** | 一般 | 高 (实际业务) | 冲突时优先私域 |
| **时效性** | 可能滞后 | 最新实践 | 私域优先 |
| **特异性** | 通用 | 企业特色 | 保留特异部分，单独标注 |

## 六、用户画像挖掘 - 增量式特征聚合

### 6.1 核心设计理念

**问题**: 
- 传统批量处理需要一次性加载所有对话，内存占用高
- 用户特征统计需要全局视图，难以分治
- 聚类分析计算量大，耗时长

**解决方案**: 增量式特征聚合 + 延迟聚类

### 6.2 技术架构

```
增量式用户画像挖掘流程:

Step 1: DataAnalyst 单通对话处理 (并行)
  ┌─────────────┐
  │ 对话日志 1   │→ 特征提取 → Tags + Behaviors + Metadata
  ├─────────────┤
  │ 对话日志 2   │→ 特征提取 → Tags + Behaviors + Metadata
  ├─────────────┤
  │ ...         │
  ├─────────────┤
  │ 对话日志 N   │→ 特征提取 → Tags + Behaviors + Metadata
  └─────────────┘
       ↓
Step 2: 增量聚合到全局统计表 (实时更新)
  global_feature_stats = {
    "price_sensitive": {count, behaviors[], dialogues[]},
    "vip_customers": {count, behaviors[], dialogues[]},
    ...
  }
       ↓
Step 3: 检查点保存 (每 10 个批次)
  save_checkpoint(global_feature_stats)
       ↓
Step 4: 所有对话处理完成后，执行聚类分析
  - 基于 global_feature_stats 进行 DBSCAN/K-Means 聚类
  - 生成用户分群和典型画像
       ↓
Step 5: 映射到 Parlant 配置
  - 生成 agent_user_profiles.json
```

### 6.3 关键实现要点

#### (1) 增量聚合算法

```python
class UserFeatureAggregator:
    def __init__(self):
        self.global_stats = defaultdict(lambda: {
            'count': 0,
            'behaviors': [],
            'dialogues': [],
            'metadata_examples': []
        })
    
    def update(self, dialogue_features):
        """实时更新全局统计"""
        for tag in dialogue_features.tags:
            self.global_stats[tag]['count'] += 1
            self.global_stats[tag]['behaviors'].extend(
                dialogue_features.behaviors
            )
            self.global_stats[tag]['dialogues'].append(
                dialogue_features.dialogue_summary
            )
            if dialogue_features.metadata:
                self.global_stats[tag]['metadata_examples'].append(
                    dialogue_features.metadata
                )
    
    def get_cluster_input(self, min_count=10):
        """获取聚类的输入数据 (过滤低频特征)"""
        return [
            {'tag': tag, **data}
            for tag, data in self.global_stats.items()
            if data['count'] >= min_count
        ]
```

#### (2) 检查点机制

```python
def save_checkpoint(aggregator, batch_num):
    """每 10 个批次保存一次检查点"""
    if batch_num % 10 == 0:
        checkpoint_path = f"checkpoints/user_features_batch_{batch_num}.pkl"
        with open(checkpoint_path, 'wb') as f:
            pickle.dump(aggregator.global_stats, f)
        print(f"✅ 保存检查点：{checkpoint_path}")

def load_latest_checkpoint():
    """恢复最新的检查点"""
    checkpoints = glob("checkpoints/user_features_batch_*.pkl")
    if checkpoints:
        latest = max(checkpoints)
        with open(latest, 'rb') as f:
            return pickle.load(f)
    return None
```

#### (3) 延迟聚类策略

**为什么延迟聚类？**
- 增量聚合阶段只需要简单的计数和追加操作，计算量小
- 聚类分析需要全局矩阵运算，计算量大
- 将聚类放在最后，避免重复计算

**聚类时机**:
- 所有对话处理完成后
- 或达到预设阈值 (如每处理 1000 通对话)

### 6.4 性能预估

假设条件:
- 对话数量：1000 通
- 单通对话特征提取时间：5 秒
- 并发数：10
- 聚类分析时间：3 分钟

**传统批量处理方案**:
```
总时间 = 1000 通 × 5 秒 ÷ 10 并发 = 8.3 分钟
聚类时间 = 约 5 分钟 (一次性加载所有数据)
总计：~13.3 分钟
内存占用 = 所有对话数据 + 特征矩阵 (可能 OOM)
```

**增量式方案**:
```
总时间 = 1000 通 × 5 秒 ÷ 10 并发 = 8.3 分钟
增量聚合时间 = ~0.5 分钟 (简单的字典操作)
检查点保存时间 = ~0.2 分钟
最终聚类时间 = ~3 分钟
总计：~12 分钟

加速比：13.3/12 = 1.1x (时间提升不明显)
内存优化：**降低 90%+**(不需要一次性加载所有对话)
```

**如果对话数量为 10000 通**:
- 传统方案：~133 分钟 (2.2 小时), 内存可能 OOM
- 增量方案：~110 分钟 (1.8 小时), 内存稳定
- 加速比：**1.2x**, 且内存占用大幅降低

### 6.5 与私域对话处理的协同

UserPortraitMiner 与 DataAnalyst 的协作关系:

```
DataAnalyst 工作流程:
  单通对话处理 → 事件提取 → 场景识别 → 流程挖掘
                          ↓
                   标注用户特征
                          ↓
              传递给 UserPortraitMiner
                          ↓
UserPortraitMiner 工作流程:
  接收标注对话 → 特征提取 → 增量聚合 → 聚类分析 → 画像生成
```

**数据流**:
```
原始对话日志
    ↓
DataAnalyst(单通处理)
    ↓
{events: [...], scenes: [...], user_tags: [...], behaviors: [...]}
    ↓
UserPortraitMiner(增量聚合)
    ↓
global_feature_stats → clustering → personas → agent_user_profiles.json
```

## 七、Agent 开发策略 - Manus 经验

### 4.1 上下文管理 (参考 Manus)

**核心原则**: 文件系统 + Git 管理，避免上下文超长

**标准文件结构**:
```
project_root/
├── task_plan.md          # 全局计划 (类似 Manus 的 todo.md)
├── progress.md           # 进度日志 (append-only)
├── findings.md           # 关键发现和决策
├── coordination_log.md   # Agent 间协调记录
└── agents/
    ├── coordinator/
    │   ├── decisions.md  # Coordinator 决策记录
    │   └── conflicts.md  # 冲突仲裁记录
    ├── data_analyst/
    │   ├── conversation_checkpoints/  # 对话处理检查点
    │   │   ├── checkpoint_batch_10.pkl
    │   │   └── checkpoint_batch_20.pkl
    │   └── private_knowledge_v1.md
    └── ...
```

**子任务输出管理**:
- 每个子任务的输出写入独立文件
- 使用 Git 进行版本管理
- 主任务只读取关键摘要，避免加载全部细节

### 4.2 防止任务偏离主题

**问题**: 子任务容易偏离原始目标 ("lost in the middle"效应)

**解决方案**: 在每个任务点都告知主题目标

**实施要点**:
- 每次 LLM 调用前包含任务计划
- 确保任务计划在 attention window 内 (最近的位置)
- 建立检测并纠正偏离的机制

**效果**:
```
上下文结构:
[原始目标 - 远端，容易被遗忘]
...大量工具调用...
[最近读取的 task_plan.md - 获得注意力！]
[当前决策点]
```

### 4.3 实施检查清单

在开发每个 Agent 时，检查以下要点:

**上下文管理**:
- [ ] 是否使用文件系统存储长期记忆
- [ ] 是否避免在上下文中存储大量原始数据
- [ ] 是否定期压缩旧数据 (只保留引用)
