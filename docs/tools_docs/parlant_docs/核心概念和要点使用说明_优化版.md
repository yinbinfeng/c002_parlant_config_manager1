## 一、Parlant 概述

### 1.1 什么是 Parlant？

Parlant 是一个开源的**AI 行为建模（ABM）引擎**，用于精确控制 LLM 智能体如何与用户交互。它通过结构化的行为模型（指南、旅程、工具等）来动态适配用户的自然交互模式，同时遵守业务规则。

### 1.2 为什么需要 Parlant？

**核心问题**：LLM 具有概率性行为的特性，相同的输入可能产生不同的响应。这导致难以预测和控制智能体的行为。

**Parlant 的解决方案**：
- **Granular Guidelines（细粒度指南）**：将行为规则分解为独立的条件 - 动作对
- **动态过滤**：自动选择最相关的指南注入上下文，避免提示词混乱
- **强制执行**：通过 Attentive Reasoning Queries (ARQs) 确保指南被遵循
- **可解释性**：每条指南的匹配和决策都可追溯

### 1.3 设计理念对比

| 方法 | 适应性 | 可预测性 | 适用场景 |
|------|--------|----------|----------|
| **Parlant (ABM)** | 高 | 高 | 需要严格遵守业务规则的客户交互 |
| Flow Engines (Rasa 等) | 低 | 高 | 严格脚本化流程 |
| Prompt Engineering | 高 | 低 | 自由对话，一致性要求低 |
| LangChain/LangGraph | 高 | 中 | 复杂任务编排 |

**核心优势**：Parlant 在高适应性和高可预测性之间取得平衡，适合生产环境的客户交互。

## 二、核心特性总览

| 特性 | 作用 | 使用时机 | 示例 |
|------|------|----------|------|
| **Agents** | 定制的 AI 人格，与用户交互的单一实体 | 始终需要 | 技术支持专家 Hexon、旅行顾问 Walker |
| **Guidelines** | 条件 - 动作对的行为规则 | 定义所有交互场景 | "当客户询问退货时，解释退货政策" |
| **Observations** | 检测对话状态，不直接行动 | 感知环境，触发其他元素 | "客户变得沮丧" |
| **Journeys** | 多轮 SOP 流程，定义状态转换 | 复杂多步骤流程 | 预约 booking、航班预订 |
| **Tools** | 外部 API 调用 | 需要获取数据或执行操作 | 查询库存、预约时间 |
| **Relationships** | 指南间的依赖/排除关系 | 处理冲突和优先级 | 排除、依赖、后产、消歧义 |
| **Canned Responses** | 预审批的回复模板 | 合规关键场景 | 法律披露、敏感话题 |
| **Glossary** | 领域专用术语定义 | 行业特定词汇 | "Ocean View = 直面大西洋的房间" |
| **Variables** | 存储动态数据 | 个性化、上下文记忆 | subscription_plan = "enterprise" |
| **Tags** | 分类跟踪对话事件 | 分析和路由 | "upsell_attempt", "human_handoff" |
| **Preamble** | 对话开场白 | 保持对话自然 | "让我帮你查查" |

**关键洞察**：这些组件共同构成行为模型（Behavior Model），像员工的培训手册一样指导 AI 行为。

## 三、快速开始

### 3.1 最小示例

```python
import parlant.sdk as p
import asyncio

async def main():
    async with p.Server() as server:
        # 创建智能体
        agent = await server.create_agent(
            name="Helper",
            description="友好的客服助手"
        )
        
        # 添加指南
        await agent.create_guideline(
            condition="客户询问价格",
            action="提供当前价格并询问具体需求"
        )
        
        print("Agent 已启动，访问 http://localhost:8800 进行测试")

asyncio.run(main())
```

### 3.2 运行步骤
1. 安装 Parlant: `pip install parlant`
2. 启动服务器：运行上述代码
3. 访问 UI: http://localhost:8800
4. 创建会话并与代理交互

## 四、核心概念详解

### 4.1 Agents（智能体）

**定义**：Parlant 中，Agent 是与用户交互的定制化 AI 人格，是"对话的对象"。

**关键特性**：
- **单一实体**：一个 Agent 是一个连贯的人格，不是多 Agent 系统
- **可定制**：每个 Agent 有自己的风格、语气、专业知识
- **独立配置**：不同部门可以有各自的 Agent（如 IT 支持、客服、销售）

**创建示例**：
```python
agent = await server.create_agent(
    name="Hexon",
    description="Technical support specialist who combines deep technical knowledge "
                "with patient explanation. Communicates with a warm, approachable tone."
)
```

**最佳实践**：
- ✅ 像招聘员工一样精心设计 Agent 的身份和人设
- ✅ 描述要具体：包含沟通风格、专业领域、价值观
- ❌ 避免过于机械的描述，加入人性化的特质

📝 **详细文档**: [`docs/concepts/entities/agents.md`](docs/concepts/entities/agents.md)

**设计建议**：Parlant 推荐将 AI Agent 建模为真人 Agent 的工作方式——如果现实中是单一角色，就用单个 AI Agent。管理复杂性比多 Agent 系统更容易。

---

### 4.2 Guidelines（指南）

**定义**：指南是行为模型的基本单元，格式为**条件 - 动作对**。

**工作原理**：
```
当 [条件] 满足时 → 执行 [动作]
```

**语法**：
```python
await agent.create_guideline(
    condition="客户使用财务术语如 DTI 或摊销",
    action="用技术深度回应——跳过基础解释",
    description="可选：提供背景信息",  # 帮助理解为什么要这样做
    criticality=p.Criticality.MEDIUM,   # 可选：LOW/MEDIUM/HIGH
    labels=["expert_mode"]              # 可选：用于分析
)
```

#### 4.2.1 指南的三种时长类型

引擎根据措辞自动推断时长：

| 类型 | 格式 | 行为 | 示例 |
|------|------|------|------|
| **一次性** | "当 X 时，做 Y" | 执行一次即完成 | "当客户点含过敏原的菜品，警告他们" |
| **持久性** | "当 X 时，做 Y 直到 W 发生" | 持续执行直到解决 | "在客户确认前，反复确认过敏原" |
| **连续性** | "只要 X，就做 Y" | 整个会话持续有效 | "只要客户是素食者，只推荐素食" |

**示例对比**：
```python
# 一次性：警告一次即可
await agent.create_guideline(
    condition="客户点了含过敏原的菜品",
    action="警告他们这道菜含有过敏原"
)

# 持久性：需要客户确认才完成
await agent.create_guideline(
    condition="客户点了含过敏原的菜品",
    action="在继续之前与他们确认是否对过敏原没问题"
)

# 连续性：只要条件成立就持续
await agent.create_guideline(
    condition="客户表示是素食者",
    action="只推荐素食菜品"
)
```

#### 4.2.2 关键性级别 (Criticality)

告诉 Agent 在匹配指南时应投入多少注意力资源：

| 级别 | 描述 | 使用时机 | 示例 |
|------|------|----------|------|
| **LOW** | 努力但不消耗大量资源 | 风格偏好、非关键推动 | "对话开始时问候客户" |
| **MEDIUM** (默认) | 更加努力，保持灵活性 | 应该有的一般规则 | "客户退货时建议换货" |
| **HIGH** | 尽最大努力，可能显得吹毛求疵 | 关键时刻、合规要求 | "检测到欺诈嫌疑时的处理" |

```python
# HIGH - 关键问题
await agent.create_observation(
    condition="客户询问取消费用",
    criticality=p.Criticality.HIGH
)

# MEDIUM - 标准业务逻辑
await agent.create_guideline(
    condition="客户想退货",
    action="建议 store credit 作为退款的替代方案",
    criticality=p.Criticality.MEDIUM
)

# LOW - 风格偏好
await agent.create_guideline(
    condition="对话刚开始",
    action="询问是否想了解促销商品",
    criticality=p.Criticality.LOW
)
```

**实用建议**：
- 默认为 MEDIUM，根据观察调整
- HIGH 保留给：合规要求、法律义务、安全检查
- LOW 用于：风格偏好和非关键行为（也有助于降低延迟和成本）

#### 4.2.3 常见陷阱

| 错误类型 | 问题 | 修正方法 |
|----------|------|----------|
| 把持续需求说成一次性 | "警告过敏原"→Agent 警告后就离开 | 改为"与他们确认" |
| 把一次性说成连续 | "总是提醒退货政策"→重复多次很烦人 | 改为"提及退货政策" |

**检查清单**：
- ✅ 动作是否需要客户响应？→ 用"确认"而非"告知"
- ✅ 是一次性信息还是持续约束？
- ✅ 关键性级别是否合适？

📝 **详细文档**: [`docs/concepts/customization/guidelines.md`](docs/concepts/customization/guidelines.md)

### 4.3 Observations（观测）

**定义**：Observation 是对话状态的检测器，它只感知不行动，用于触发其他元素（指南、工具等）。

**与 Guidelines 的区别**：
- **Observation**：仅检测状态，不包含动作
- **Guideline**：包含条件 + 动作

**语法**：
```python
# 创建观测
customer_suspects_fraud = await agent.create_observation(
    condition="客户怀疑卡上有未经授权的交易",
    criticality=p.Criticality.HIGH
)

# 观测可以触发工具
await agent.create_guideline(
    condition="客户想对交易采取行动",
    action="询问他们是要争议交易还是锁定卡片",
    dependencies=[customer_suspects_fraud],  # 只有观测触发时才激活
    tools=[freeze_card_tool]
)
```

**使用场景**：
1. **门控指南**：Observation 检测广泛状态，Guideline 在该上下文中生效
2. **排除冲突**：当 Observation 触发时，排除不相关的指南
3. **工具触发**：仅在特定观测匹配时激活工具

**示例模式**：
```python
# 模式 1: 观测作为门
customer_wants_deals = await agent.create_observation(
    condition="客户表达对特价商品或折扣的兴趣"
)

offer_discount = await agent.create_guideline(
    condition="有活跃的季节性促销",
    action="提及当前的季节性折扣",
    dependencies=[customer_wants_deals]  # 只在客户感兴趣时触发
)

# 模式 2: 观测排除指南
customer_is_upset = await agent.create_observation(
    condition="客户变得沮丧或沮丧"
)

upsell_premium = await agent.create_guideline(
    condition="客户正在浏览产品",
    action="推荐高级升级",
)

# 客户沮丧时不追加销售
await customer_is_upset.exclude(upsell_premium)
```

📝 **详细文档**: [`docs/concepts/customization/guidelines.md`](docs/concepts/customization/guidelines.md)

### 4.4 Journeys（旅程）

**定义**：Journey 是多轮 SOP（标准作业程序），为预订、故障排除或入职等流程定义流程。Agent 遵循流程但会适应——它可以快进状态、回顾早期状态，或根据客户互动方式调整节奏。

**核心价值**：
- 结构化复杂的多轮交互
- 允许分支和条件路径
- 自动管理上下文和状态转换
- 支持工具调用和数据加载

**基本结构**：
```python
journey = await agent.create_journey(
    title="预订航班",
    description="引导客户完成航班预订",
    conditions=["客户想预订航班"],
)

# 初始状态
t0 = await journey.initial_state.transition_to(
    chat_state="询问目的地和出行日期"
)

# 工具状态：加载可用航班
t1 = await t0.target.transition_to(
    tool_state=get_available_flights
)

# 聊天状态：展示航班选项
t2 = await t1.target.transition_to(
    chat_state="列出可用航班并询问选择"
)

# 条件分支：客户选择航班
t3 = await t2.target.transition_to(
    chat_state="收集乘客信息并确认预订详情",
    condition="客户选择了航班"
)

# 工具状态：执行预订
t4 = await t3.target.transition_to(
    tool_state=book_flight,
    condition="客户确认预订详情"
)

# 结束状态
t5 = await t4.target.transition_to(
    chat_state="提供确认号和预订摘要"
)
await t5.target.transition_to(state=p.END_JOURNEY)
```

#### 4.4.1 状态类型

| 状态类型 | 作用 | 示例 |
|----------|------|------|
| **Chat State** | Agent 在引导下与客户对话 | "询问客户来自哪里" |
| **Tool State** | Agent 调用工具并加载结果到上下文 | `get_available_slots` |
| **Fork State** | 路由状态，评估条件并分支流 | 年龄验证后的分支 |

**状态转换规则**：
- Tool State → Chat State：自动将工具结果加载到上下文
- 不建议 Tool State → Tool State：会产生明显延迟
- 最佳实践：一个 Tool State 执行所有必要操作，然后 Chat State 展示结果

#### 4.4.2 转换类型

**直接转换**：始终进行，无分支
```python
t = await state.transition_to(chat_state="询问客户姓名")
```

**条件转换**：仅当条件满足时进行
```python
t = await state.transition_to(
    chat_state="提供高级套餐",
    condition="客户超过 21 岁且来自美国或加拿大"
)
```

**跳转到现有状态**：合并路径
```python
t = await state.transition_to(state=existing_state)
t = await state.transition_to(state=existing_state, condition=CONDITION)
```

#### 4.4.3 分叉状态 (Fork States)

当需要从多个状态以相同方式分支时使用：

```python
# 收集信息
t1 = await journey.initial_state.transition_to(
    chat_state="询问客户来自哪里"
)

t2 = await t1.target.transition_to(
    chat_state="询问客户年龄"
)

# 创建分叉点
fork = await t2.target.fork()

# 基于条件的三个分支
t3 = await fork.target.transition_to(
    chat_state="提供高级品酒套餐",
    condition="客户超过 21 岁且来自美国或加拿大"
)
t4 = await fork.target.transition_to(
    chat_state="提供国际体验套餐",
    condition="客户超过 21 岁但不是来自美国/加拿大"
)
t5 = await fork.target.transition_to(
    chat_state="提供家庭友好活动套餐",
    condition="客户未满 21 岁"
)
```

**何时使用分叉**：
- ✅ 需要在 Tool State 后分支
- ✅ 多个状态需要扇出到相同的条件分支
- ❌ 简单的聊天状态分支可以用条件转换代替

2. 复制生成的 Mermaid 代码

### 4.5 Tools（工具）

**定义**：Tools 是外部 API 和工作流的封装，仅在 Observation 匹配时触发，避免误调用。

**核心优势**：
- **按需激活**：仅在相关上下文中可用，不会长期停留在语境中
- **减少误报**：传统 LLM 工具调用常有的问题
- **参数安全**：明确的参数验证和类型检查

**创建工具**：
```python
@p.tool
async def get_insurance_providers(context: p.ToolContext) -> p.ToolResult:
    return p.ToolResult(["Mega Insurance", "Acme Insurance"])

@p.tool
async def schedule_appointment(context: p.ToolContext, datetime: datetime) -> p.ToolResult:
    # 模拟预约
    return p.ToolResult(data=f"Appointment scheduled for {datetime}")
```

**使用工具**：
```python
# 方式 1: 直接在 Guideline 中使用
await agent.create_guideline(
    condition="客户询问保险",
    action="列出我们接受的保险提供商，并告诉他们致电办公室了解更多",
    tools=[get_insurance_providers]
)

# 方式 2: 通过 Observation 触发
await agent.create_observation(
    condition="客户询问服务功能",
    tools=[query_docs_tool]
)
```

**工具最佳实践**：
- ✅ 工具应该幂等：多次调用产生相同结果
- ✅ 错误处理清晰：返回有意义的错误消息
- ✅ 参数明确：使用类型注解和文档字符串
- ✅ 响应格式化：使用 `p.ToolResult` 包装数据

📝 **详细文档**: [`docs/concepts/customization/tools.md`](docs/concepts/customization/tools.md)

### 4.6 Relationships（关系）

**定义**：Relationships 定义元素（Guidelines、Observations、Journeys）之间的依赖和排除关系，保持上下文狭窄和聚焦。

#### 4.6.1 关系类型总览

| 关系 | 语法 | 效果 | 使用场景 |
|------|------|------|----------|
| **排除** | `source.exclude(target)` | S 和 T 同时激活时，只有 S 激活 | 防止指令冲突，控制优先级 |
| **依赖** | `source.depend_on(target)` | S 激活时，除非 T 也激活否则关闭 | 观察导向，基线条件 |
| **后产** | `source.entail(target)` | S 激活时，T 也应该始终激活 | 链式动作，避免时间问题 |
| **消歧义** | `source.disambiguate([T1, T2])` | S 激活且多个 T 激活时，请客户澄清 | 模糊意图澄清 |

#### 4.6.2 排除关系 (Exclude)

**作用**：当 S 和 T 同时激活时，只有 S 应被激活。

**使用场景**：
1. **Observation 排除 Guideline**：
```python
customer_is_upset = await agent.create_observation(
    condition="客户变得沮丧"
)

upsell_premium = await agent.create_guideline(
    condition="客户正在浏览产品",
    action="推荐高级升级"
)

# 客户沮丧时不追加销售
await customer_is_upset.exclude(upsell_premium)
```

2. **Guideline 排除 Guideline**：
```python
handoff_if_upset = await agent.create_guideline(
    condition="客户非常沮丧",
    action="转接给人工客服"
)

offer_pepsi = await agent.create_guideline(
    condition="客户要可乐",
    action="提供百事可乐代替可口可乐"
)

# 优先处理客户情绪而非饮料偏好
await handoff_if_upset.exclude(offer_pepsi)
```

3. **控制优先级顺序**：
```python
complete_transaction = await agent.create_guideline(
    condition="客户想进行交易",
    action="引导他们完成整个流程"
)

offer_savings = await agent.create_guideline(
    condition="客户账户余额少于$1,000",
    action="提供储蓄计划"
)

# 不要在交易过程中打断
await complete_transaction.exclude(offer_savings)
```

**别名**：`prioritize_over` 与 `exclude` 效果相同，但语义更直观
```python
await source.prioritize_over(target)  # 等同于 exclude
```

#### 4.6.3 依赖关系 (Dependency)

**作用**：当 S 激活时，除非 T 也被激活，否则关闭它。

**使用场景**：
1. **Observation 门控 Guideline**：
```python
customer_wants_deals = await agent.create_observation(
    condition="客户表达对特价商品或折扣的兴趣"
)

offer_seasonal_discount = await agent.create_guideline(
    condition="有活跃的季节性促销",
    action="提及当前的季节性折扣",
    dependencies=[customer_wants_deals]  # 只在客户感兴趣时触发
)

offer_loyalty_reward = await agent.create_guideline(
    condition="客户已成为会员超过一年",
    action="提供他们的忠诚度奖励",
    dependencies=[customer_wants_deals]
)
```

2. **边缘情况依赖于基线**：
```python
handle_return = await agent.create_guideline(
    condition="客户想退货",
    action="帮助他们完成退货流程"
)

no_order_number = await agent.create_guideline(
    condition="客户无法提供订单号",
    action="加载他们最后一个订单的商品并要求确认",
    dependencies=[handle_return]  # 只在退货流程中生效
)

has_order_number = await agent.create_guideline(
    condition="客户指定了确切订单号",
    action="加载该订单的商品并要求确认",
    dependencies=[handle_return]
)
```

#### 4.6.4 后产关系 (Entailment)

**作用**：当 S 激活时，T 也应该始终被激活。

**解决的问题**：Parlant 在撰写回复前评估所有指南。如果 Guideline A 的动作会让 Agent 做出 Guideline B 条件依赖的行为，B 就不会匹配——因为评估时该行动还没发生。

**示例**：
```python
explain_returns = await agent.create_guideline(
    condition="客户询问退货",
    action="解释退货政策"
)

mention_warranty = await agent.create_guideline(
    condition="Agent 正在解释退货政策",
    action="同时提及 30 天保修"
)

# 没有 entailment 时，mention_warranty 不会触发
# 因为评估时 Agent 还没解释退货政策
await explain_returns.entail(mention_warranty)
```

**效果**：保证 mention_warranty 在 explain_returns 触发时自动触发，无需依赖预测条件匹配。

#### 4.6.5 消歧义关系 (Disambiguation)

**作用**：当 S 激活且激活两个或多个目标时，请客户澄清他们想采取哪种行动。

**使用场景**：
```python
fetch_atm_limits = await agent.create_guideline(
    condition="客户询问 ATM 限额",
    action="从他们的账户档案中获取数据"
)

fetch_credit_card_limits = await agent.create_guideline(
    condition="客户询问信用卡限额",
    action="从卡片提供商处获取"
)

ambiguous_limits = await agent.create_observation(
    condition="客户询问限额但不清楚是哪种"
)

# 让客户澄清是 ATM 限额还是信用卡限额
await ambiguous_limits.disambiguate([fetch_atm_limits, fetch_credit_card_limits])
```

**效果**：如果客户问"我的限额是多少？"且两条指南一致，Agent 会询问具体指的是哪种限额，而不是随意选一个。

📝 **详细文档**: [`docs/concepts/customization/relationships.md`](docs/concepts/customization/relationships.md)

### 4.7 Canned Responses（预设回复）

**定义**：预先批准的回复模板，在关键时刻消除幻觉风险，保持措辞准确。

**使用场景**：
- 合规披露、法律义务
- 敏感话题（医疗建议、财务建议）
- 品牌一致性要求高的场景

#### 4.7.1 组合模式 (Composition Modes)

| 模式 | 行为 | 使用场景 |
|------|------|----------|
| **Fluid** | 优先从预设回复中选择，找不到好的匹配则回退到默认生成 | (A) 大部分流畅，但控制特定场景<br>(B) 原型开发，逐步生成预设回复 |
| **Composited** | 仅使用预设回复候选来修改生成的草稿消息，模仿其风格 | 品牌敏感场景，语气很重要 |
| **Strict** | 只能输出提供的回复。如无匹配，发送可自定义的 no-match 消息 | 高风险场景，不能有任何幻觉 |

#### 4.7.2 使用示例

```python
await agent.create_guideline(
    condition="客户讨论与我们业务无关的事情",
    action="告诉他们你无法帮助",
    # Strict 模式：完全消除幻觉
    composition_mode=p.CompositionMode.STRICT,
    canned_responses=[
        await agent.create_canned_response(
            "抱歉，我无法帮助您处理该问题。"
        )
    ],
    priority=100,  # 最高优先级
)
```

**最佳实践**：
- ✅ 为关键场景准备多个变体，提高匹配率
- ✅ 定期审查和更新预设回复
- ✅ 结合 Tags 分析哪些预设回复最有效

📝 **详细文档**: [`docs/concepts/customization/canned-responses.md`](docs/concepts/customization/canned-responses.md)

### 5.1 匹配处理器 (Match Handlers)

**定义**：Handlers 是在旅程或单个状态上注册的回调函数，用于触发日志记录、分析或外部集成等副作用。

#### 5.1.1 旅程级处理器

```python
async def on_journey_activated(ctx: p.EngineContext, match: p.JourneyMatch) -> None:

        "customer": ctx.customer.id,

)
```

**使用场景**：
- 分析追踪（tracking conversion rates）
- 外部系统集成（CRM 更新）

### 5.2 消歧义处理

**问题**：客户的话可能被多种解读，导致对采取何种行动感到困惑。

**解决方案**：使用 `disambiguate()` 让 Agent 在可以采取多种行动时，要求客户澄清意图。

**完整示例**：
```python
async def main() -> None:
    async with p.Server() as server:
        agent = await server.create_agent(
            name="Healthcare Agent",
            description="Is empathetic and calming to the patient."
        )
        
        scheduling_journey = await create_scheduling_journey(server, agent)
        lab_results_journey = await create_lab_results_journey(server, agent)
        
        # 创建模糊情况的 observation
        status_inquiry = await agent.create_observation(
            "患者询问复诊，但不清楚是哪种方式"
        )
        
        # 使用 disambiguate 在两个旅程之间澄清
        await status_inquiry.disambiguate([
            scheduling_journey, 
            lab_results_journey
        ])

asyncio.run(main())
```

**效果**：如果患者以含糊的方式询问复诊（如"我需要跟进上次就诊"），Agent 会要求他们澄清是要预约还是查看化验结果。

**最佳实践**：
- ✅ 为模糊意图创建明确的 observation
- ✅ 列出所有可能的解读（Journeys 或 Guidelines）
- ✅ 措辞礼貌且专业

📝 **详细文档**: [`docs/concepts/customization/relationships.md#disambiguation`](docs/concepts/customization/relationships.md)

### 5.3 客户群体特定指南

**场景**：需要为特定客户群体（如 VIP、企业客户）提供差异化服务。

**实现方式**：使用自定义 Matcher 绕过默认的 LLM 匹配，使用确定性逻辑。

**完整示例**：
```python
# 1. 创建 VIP 标签
vip_tag = await server.create_tag(name="VIP")

# 2. 定义基于代码的 matcher
async def is_vip(
    ctx: p.GuidelineMatchingContext,
    guideline: p.Guideline
) -> p.GuidelineMatch:
    matched = vip_tag.id in p.Customer.current.tags
    
    return p.GuidelineMatch(
        id=guideline.id,
        matched=matched,
        rationale="Customer has VIP tag" if matched else "Customer does not have VIP tag"
    )

# 3. 使用该 matcher 创建 observation
vip_observation = await agent.create_observation(matcher=is_vip)

# 4. 创建仅对 VIP 客户激活的指南
vip_guideline = await agent.create_guideline(
    condition="客户询问价格",
    action="提供独家 VIP 折扣 20%",
    dependencies=[vip_observation]  # 关键：只有 VIP 才激活
)
```

**关键点**：
- `matcher` 参数允许使用确定性逻辑而非 LLM 匹配
- `dependencies` 确保指南只在 observation 匹配时激活
- 非 VIP 客户永远不会看到 VIP 专属行为

**扩展应用**：
- 企业客户专属支持
- 地区特定促销
- 会员等级差异化服务

📝 **详细文档**: [`docs/concepts/entities/customers.md#segment-specific-guidelines`](docs/concepts/entities/customers.md)

## 六、最佳实践与注意事项

### 6.1 指南设计原则

#### 6.1.1 动作简短，用描述详细说明

**原理**：
- Parlant 的引擎在合规检查中使用条件和动作文本
- 更长的条件和动作会增加响应延迟
- 描述只在指南匹配时才加载到上下文

**错误示例**：
```python
# ❌ 动作太长，影响匹配性能
await agent.create_guideline(
    condition="客户提到竞品",
    action="Acknowledge their experience with the competitor respectfully, "
           "never disparage them directly, and highlight our key differentiators "
           "such as 24/7 support, flexible pricing, and our satisfaction guarantee, "
           "while focusing on what makes us uniquely valuable for their use case"
)
```

**正确示例**：
```python
# ✅ 动作简短，细节放在描述中
await agent.create_guideline(
    condition="客户提到竞品",
    action="Acknowledge their experience and highlight our differentiators",
    description="Be respectful of competitor products (BrandX, ABC Corp, etc.). "
                "Never disparage them directly. Focus on our key differentiators: "
                "24/7 support, flexible pricing, and satisfaction guarantee."
)
```

**检查清单**：
- ✅ 动作是否简短明了（1-2 句话）？
- ✅ 详细描述是否在 description 字段？
- ✅ 条件是否清晰可观察？

#### 6.1.2 注意边缘情况

**问题**：LLMs 会字面理解指导。如果你告诉 Agent"总是推荐高级功能"，即使客户抱怨价格，它也可能这么做。

**解决方案**：
```python
# ❌ 过于绝对
await agent.create_guideline(
    condition="客户浏览产品",
    action="总是推荐高级功能"
)

# ✅ 考虑背景
await agent.create_guideline(
    condition="客户浏览产品且未表达对价格的担忧",
    action="推荐高级功能并强调价值"
)

await agent.create_guideline(
    condition="客户提到预算限制",
    action="推荐性价比最高的基础选项"
)
```

**建议**：宁愿模糊也不要过于具体，提供清晰的上下文指导而非脚本化 every interaction。

### 6.2 关系使用策略

#### 6.2.1 排除关系的时机

**使用排除**：
- ✅ 两条指南可能冲突时
- ✅ 需要控制对话中的先后顺序
- ✅ Observation 检测到特殊状态（如客户沮丧）

**示例**：
```python
# 客户沮丧时不进行追加销售
customer_is_upset = await agent.create_observation(
    condition="客户变得沮丧"
)

upsell = await agent.create_guideline(
    condition="客户浏览产品",
    action="推荐高级升级"
)

await customer_is_upset.exclude(upsell)
```

#### 6.2.2 依赖关系的层级

**模式**：Observation 检测广泛状态 → Guideline 在该上下文中生效

```python
# 基线观察
customer_wants_deals = await agent.create_observation(
    condition="客户表达对折扣的兴趣"
)

# 依赖于基线的具体指南
offer_seasonal = await agent.create_guideline(
    condition="有季节性促销",
    action="提及当前折扣",
    dependencies=[customer_wants_deals]
)

offer_loyalty = await agent.create_guideline(
    condition="客户是长期会员",
    action="提供忠诚度奖励",
    dependencies=[customer_wants_deals]
)
```

#### 6.2.3 后产解决时间问题

**问题**：Parlant 在撰写回复前评估所有指南。如果 Guideline A 的动作会让 Agent 做出 Guideline B 条件依赖的行为，B 就不会匹配。

**解决**：
```python
explain_policy = await agent.create_guideline(
    condition="客户询问退货",
    action="解释退货政策"
)

mention_warranty = await agent.create_guideline(
    condition="Agent 正在解释退货政策",
    action="同时提及 30 天保修"
)

# 保证 B 在 A 触发时自动触发
await explain_policy.entail(mention_warranty)
```

### 6.3 旅程设计技巧

#### 6.3.1 状态类型的选择

| 场景 | 推荐状态 | 理由 |
|------|----------|------|
| 简单对话 | Chat State | 直接在引导下交流 |
| 需要外部数据 | Tool State → Chat State | 先获取数据再呈现结果 |
| 多分支决策 | Fork State | 清晰的条件分支 |

**最佳实践**：
- ✅ Tool State 后始终跟随 Chat State（避免延迟）
- ✅ 用一个 Tool State 执行所有必要操作
- ❌ 避免连续的 Tool State

#### 6.3.2 条件转换 vs 分叉状态

**使用条件转换**（更简单）：
```python
t = await state.transition_to(
    chat_state="提供高级品酒套餐",
    condition="客户超过 21 岁且来自美国或加拿大"
)

t = await state.transition_to(
    chat_state="提供国际体验套餐",
    condition="客户超过 21 岁但不是来自美国/加拿大"
)
```

**使用分叉状态**（需要时可重用）：
```python
# 多个状态都需要相同的分支逻辑
fork = await state.fork()

t1 = await fork.target.transition_to(..., condition="...")
t2 = await fork.target.transition_to(..., condition="...")

# 其他状态也可以转移到这个 fork
await another_state.transition_to(state=fork.target)
```

### 6.4 工具和 API 设计

#### 6.4.1 工具设计原则

**DO**：
- ✅ 工具应该幂等（多次调用结果相同）
- ✅ 错误处理清晰，返回有意义的错误消息
- ✅ 参数明确，使用类型注解
- ✅ 响应使用 `p.ToolResult` 包装

**DON'T**：
- ❌ 不要假设 Agent 知道所有参数的含义
- ❌ 不要返回未格式化的原始数据
- ❌ 不要在工具中执行耗时操作（超时风险）

#### 6.4.2 参数安全

**问题**：Agent 可能猜错参数值。

**解决**：
```python
@p.tool
async def schedule_appointment(
    context: p.ToolContext,
    datetime: datetime,  # 明确要求 datetime 类型
    patient_id: str      # 从 context 获取，不要猜测
) -> p.ToolResult:
    # 验证参数
    if datetime < datetime.now():
        return p.ToolResult(error="Cannot schedule in the past")
    
    # 执行预约
    result = db.schedule(patient_id, datetime)
    return p.ToolResult(data=f"Appointment scheduled: {result}")
```

### 6.6 性能优化

#### 6.6.1 减少延迟

**策略**：
1. **简短的动作文本**：减少匹配时的计算开销
2. **合理使用 Criticality**：
   - LOW 用于非关键指南（减少资源消耗）
   - HIGH 仅用于真正关键的场景
3. **避免过多连续指南**：每条都增加评估开销

#### 6.6.2 成本控制

**策略**：
- 使用 LOW criticality 减少非关键指南的 token 消耗
- 优化 description 长度（只在匹配时加载）
- 定期清理不再使用的指南和旅程

### A.3 常见问题 FAQ

**Q: 指南为什么不触发？**
- 检查条件是否太具体或太模糊
- 查看日志中的匹配分数和 rationale
- 尝试添加 synonyms 或调整措辞

**Q: 如何让指南只触发一次？**
- 使用一次性措辞："警告他们"而非"与他们确认"
- 或使用 `track=False` 参数

**Q: Journey 卡在某个状态怎么办？**
- 检查是否有符合条件的转换
- 确保有兜底路径（edge case 处理）
- 使用可视化工具检查流程图

**Q: 如何优化响应速度？**
- 缩短条件和动作文本
- 使用 LOW criticality 用于非关键指南
- 减少不必要的 Tools

5. **Canned Responses**：预审批回复，消除关键场景幻觉
6. **Customers & Tags**：个性化和细分支持
7. **Explainability**：完整的推理日志，便于调试
